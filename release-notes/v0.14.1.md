# v0.14.1 — `/design` is back, with +29 pp of proof

A packaging fix for v0.13.0. The `/design` skill was advertised in the v0.13.0 changelog, README, and landing page — but the actual skill files were never committed. v0.14.0's folder-format migration moved the broken reference path without fixing it.

**This release restores the missing files** in v0.14 folder format, and adds a companion-installer path so any tier2 skill that ships a slash command + template pack installs cleanly via `aikit add <name>`.

## Install

```bash
npm i -g haac-aikit@0.14.1
aikit add design
```

That installs the skill (`.claude/skills/design/SKILL.md`), the slash command (`.claude/commands/design.md`), the marker-bounded starter (`.aikit/templates/design/starter-DESIGN.md`), and the interactive showroom (`.aikit/templates/design/template.html`). Then run `/design` in Claude Code to bootstrap a project-root `DESIGN.md`.

## Highlights

### `/design` skill — restored

Codify the project's visual language as a marker-bounded `DESIGN.md` contract every AI tool reads. Five stable section IDs (`atmosphere`, `colors`, `typography`, `components`, `layout`), each surgically updatable via `/design refine "<change>"` without touching the others.

Voice rules enforce:

- Hex codes in inline `` `code` `` ticks (so the showroom's color pickers can bind to them)
- No Tailwind class names or design-token aliases as the source of truth
- Full font stacks (not just "serif" or "sans")
- Numerical layout specs (no adjectival hand-waving)

### Interactive HTML showroom

A self-contained `docs/design/index.html` rendered from `DESIGN.md`:

- Live color pickers bound to CSS custom properties — change a swatch, see buttons and cards update immediately
- Font dropdowns that swap typography specimens live
- **Copy Markdown** button that serializes the current state back to clipboard for round-trip into `/design refine "<change>"`
- Light / dark toggle
- No CDN, no build step, no dependencies — opens off disk and works offline

### +29 pp benchmark

Re-validated head-to-head against the freestyle baseline on 3 scenarios, parallel subagent runs, scored against a 7-criterion binary rubric:

| Scenario | With `/design` | Freestyle | Δ |
|---|---|---|---|
| Extract from pasted HTML | 100% | 71% | +29 |
| Synthesize from a verbal brief | 100% | 71% | +29 |
| Build from a described screenshot | 86% | 57% | +29 |
| **Average** | **95%** | **66%** | **+29** |

**What the gap is.** Freestyle always wins on the *descriptive* criteria — hex codes in `code` ticks, full font stacks, numerical layout specs all appear naturally. It always loses on the *structural* contract: it invents its own section IDs (`design-overview`, `design.tokens.color`) and marker patterns (`<!-- BEGIN:design-X -->`) instead of the engine's canonical `<!-- BEGIN:haac-aikit:section:X -->`. That's the gap `/design` exists to close — and why surgical `/design refine` works afterwards.

Eval harness lives at [`evals/design/`](https://github.com/Hamad-Center/haac-aikit/tree/main/evals/design) — reproducible from the scenario files alone.

### `aikit add <name>` — now installs companions

When you add a skill that ships an opinionated companion bundle, `aikit add <name>` now copies all of it in one shot:

- `catalog/commands/<name>.md` → `.claude/commands/<name>.md`
- Every file under `catalog/templates/<name>/` → `.aikit/templates/<name>/`

Strictly additive — skills without companions still install exactly as before. Makes tier2-with-companions a first-class pattern across the catalog.

## Why this is a patch

The semver answer would arguably be a minor bump — `installSkillCompanions()` is genuinely new behavior. But this release is, in spirit, the fix for v0.13.0's broken packaging promise. The skill that was supposed to be in the box from day one is finally in the box.
